Linux系统如何收发网络包

网络模型

为了各种设备能够通过网络相互通信,解决不同设备在网络互联中的兼容性问题,制定了 开放式系统互联通信参考模型 OSI网络模型

共七层,应用层,表示层,会话层,传输层,网络层,数据链路层,以及物理层

  • 应用层: 给程序提供统一接口
  • 表示层: 数据转换成兼容另一个系统的格式
  • 会话层: 建立管理和终止表示层实体之间的通信会话
  • 传输层: 负责端到端的数据传输
  • 网络层: 负责数据的路由,转发,分片
  • 数据链路层: 负责数据的封帧和差错检测,以及MAC寻址
  • 物理层:负责在物理网络中传输数据帧

但是它过于复杂,分层也是是概念和理论,其实没有具体实现方案

实际上常见且实用的是四层模型,TCP/IP模型

  • 应用层: 提供应用程序,如HTTP,DNS,FTP等
    (http报文的响应头啊,方法啊啥的)
  • 传输层: 负责端到端的通信,TCP,UDP
    (头部含端口号,负责对应的端口)
  • 网络层: 网络包的封装,分片,路由,转发等,例如 IP, ICMP等
    (包含分片数据包,协议号,还有源ip地址和目标ip地址,实现这些内容交给了IP协议这样的东西)
  • 网络接口层: 负责网络包在屋里网络中的传输,例如 网络包的封帧, MAC寻址,差错检测,以及网卡传输网络帧等
    (这部分)

HTTP基本概念

http超文本传输协议

  • 超文本: 超越了普通文本,是混合体
  • 传输: 双向协议,是专门用来在两点之间传输数据的约定和规范,允许中间有中转或接力,只要不打扰基本数据传输
  • 协议: 使用计算机能够理解的语言确立计算机间交流通信的规范,以及相关的各种控制和错误处理方式

因此HTTP是计算机世界里专门进行[两点]之间[传输]文字图片音频等[超文本]数据的[约定和规范]

http状态码

  • 1xx : 提示信息, 协议处理的中间状态,很少使用

  • 2xx : 服务器成功处理了客户端请求

    1. [ 200 ok ] 一切正常 (非head请求会带有body数据) 看不懂,待补充
    2. [ 204 no content] 响应头没有body数据
    3. [ 206 partial content] http分块下载或断点续传,表示相应返回的body并非全部

  • 3xx : 客户端请求的资源发生了变动,需要客户端用新的url重新发送请求获取, 即 重定向

    1. [ 301 moved permanently] 永久重定向,资源已经不存在了
    2. [ 302 found ] 临时重定向,资源存在,但是需要另一个url访问

    都会在响应头使用Location字段,知名后续跳转,可以自动重定向到新的

  • 4xx : 表示客户端发送到报文有误,服务器无法处理,错误码

    1. [ 400 bad request ] 报文有错,
    2. [ 403 Forbidden ] 服务器禁止访问资源
    3. [ 404 not found ] 请求的资源在服务器上不存在未找到

  • 5xx : 报文正确,但是 服务器处理事内部发生了错误,是服务端的错误码

    1. 500 通用
    2. [ 501 not implemente] 功能还不支持
    3. [ 502 bad gateway ] 服务器作为网关或者代理时返回的错误码,自身正常,后端服务器发生错误
    4. [ 503 service unavailable ] 服务器很忙暂时无法响应

HTTP常见字段

  • Host
    制定服务器域名,可以发往 同一台 服务器的不同网站
  • content-length
    服务器返回数据时,表明本次回应的数据长度,浏览器可以以这个字段的长度找到 HTTP body 的边界 而httphead的便捷靠回车或换行——解决了”粘包” 待补充
  • connection
    用于客户端要求服务器使用 http长连接 机制, 以便其他请求复用
    只要忍一段没有明确提出断开连接,就保持TCP的链接状态
  • content-type
    服务器回应时告诉客户端,本次是什么格式;什么编码
    而客户端请求的时候可以使用 Accept 字段声明自己可以接受那些数据格式
  • content-encoding
    数据的压缩方法,表示服务器端使用了什么压缩格式
    对应客户端 accept-encoding

GET 和 POST

区别

根据 RFC规范 get和post 有 语义上的区别

  • GET : 从服务器获取指定资源, 参数写在URL中(query参数,在浏览器地址栏里面,浏览器也会对url有限制)
  • POST : 根据 请求负荷 对指定资源做出处理,(body参数) body可以是任意格式的数据

都是安全和幂等的吗?

  1. 安全 : 该请求方法不会 破坏 服务器上的资源
  2. 幂等 : 执行多次相同的操作,结果都是 相同 的

如果按照了RFC规范定义的语义:

  • GET是安全且幂等的，
    因此浏览器可以对GET请求的数据作缓存,这个缓存可以做到避免浏览器发请求,或者做到代理上 代理什么意思?? 待补充
  • POST 是不安全的，不冪等的，

RFC规范并没有规定GET请求不能带body, 但是因为语义上不需要用到body,所以基本不用

HTTP缓存技术

有哪些实现方式?

强制缓存

浏览器判断缓存没有过期,就直接使用浏览器的本地缓存—– 由浏览器决定

强缓存由http头部的两个字段实现,表示资源在客户端缓存的有效期

  • cache-control : 相对时间 优先级更高, 设置更加精细
    1. 第一次访问服务器资源师,服务器会在返回这个资源的同时,在响应头部加上这个字段,设置了过期时间的大小
    2. 再次访问时, 浏览器会根据请求资源的时间和时间大小计算出是否过期,过期重新请求
    3. 再次收到后就更新
  • expires : 绝对时间

协商缓存

某些请求的响应码是 304 , 这个是告诉浏览器 可以使用本地缓存的资源 , 这种通过服务端告知客户端是否可以使用缓存的方式被称为协商缓存

协商缓存基于两种头部来实现

第一种 请求头部中的 If-Modified-Since 字段 和响应头部中的 Last-Modified 字段实现

  • last-modified : 标志着这个响应资源的最后修改时间, (服务端)
  • fi-modified-since : 当资源过期了,发现响应头中有 (之前返回的时候带着的) 最后修改时间的声明 ,那么发请求的时候带上这个 最后修改时间 (放在这个 是否修改时间 if这个字段里面) 服务器发现有这个字段之后,与最后的修改时间进行对比,如果被改动过,就返回最新的,如果没修改,那么就响应 http 304 走缓存

第二种 请求头中的 If-None-Match 和 响应头中的 ETag 字段

  • if -none-match : 当资源过期时,浏览器发现响应头里有Etag,再次向服务器发起请求,会将这个值设置成etag的值,服务器收到请求后进行对比,如果资源没有变化就返回304,如果资源变化了就返回200
  • etag : 唯一标识响应资源

后者是基于一个唯一标识实现,比起时间更为准确,因此etag的优先级更高

因为它能够解决几个时间难以处理的问题:

  1. 没有修改文件内容的情况下,时间也可能改变
  2. 有些文件是秒级以内修改,但是最后修改时间现查不到
  3. 有的服务器不能精确获取最后修改文件时间

都需要配合cache-control 使用,未命中强制缓存时, ( 即过期时间已经超过了 ) 这时候才会发起带有协商缓存字段的请求

HTTP特性

HTTP/1.1 的优点

  1. 简单 : 基本报文格式是 header +body 头部信息是 key-value 的简单文本形式,易于理解
  2. 灵活和易于扩展 : 各类请求方法, URI/URL, 状态码 , 头字段没有被固定死,允许自定
    作用在应用层,下层可以随意变化
    • HTTPS 就是在http和tcp之间增加了ssl/tsl安全传输层
    • 而http/1.1 http/2.0 使用tcp协议, 3.0改用udp协议
  3. 应用广泛 可以跨平台
