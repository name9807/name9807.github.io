协议栈

协议栈的内部分成多个部分,上级会委托下级工作

应用程序 调用 socket库 来委托协议栈工作

上半部分:TCP UDP
下班部分:ip协议控制收发操作(包含icmp协议 和 arp 协议)

  • icmp协议 告知网络包传送过程中错误和控制信息
  • ARP 根据 IP 查询

TCP

tcp报文的头部

  • 源端口号和目标端口号—知道发给哪个应用
  • 序号 —– 解决包的乱序
  • 确认号 —— 没收到就重新发送,解决丢包问题
  • 状态位—- ( 首部长度 后的保留位 )
    • syn 链接
    • ack 回复
    • rst 重新连接
    • fin 结束
  • 窗口大小—双方各自标识的能够的处理能力

3次握手—– 建立链接

建立连接的过程中, 双方的状态位的时序图如下

一切开始于服务端主动监听某个端口 处于listen状态

  • 当客户端发起连接 发送 syn
  • 服务端收到 链接 返回了 syn 并且对这个syn ack
  • 客户端收到 syn 和 ack 之后 发送 对syn 的ack 的 ack
  • 服务端收到之后—–链接成功建立

切割数据

如果超过了 mss 的长度, 这时候 TCP 会把 http 数据拆解成一块块的数据

  • MTU 一个网络包的最大长度
  • MSS 除去头部之后,一个网络包能容纳的TCP数据的最大长度

生成报文

建立连接之后, 进行分包之后, 每一个数据包(tcp段)的数据部分 存放了http头 + 数据

**至此,http报文 在传输层的处理结束 实现了可靠传输 **

IP

实际上, tcp 模块 在执行链接,收发,断开 这些阶段的操作时,都需要委托 IP 模块去封装成网络包 来发送

上文中的链接 syn ack 等等的收发 都已经通过ip协议 发送过

IP报文头部格式如下:

  • 原地址IP 和 目标地址IP :客户端输出的 IP 地址和 通过 DNS 域名解析得到的服务器IP
  • 协议号:用协议号标记经过什么协议传输,本例子中http经过tcp传输,所以协议号06(十六进制) —- 表示tcp
  • 源地址IP选择哪一个网卡的地址,需要根据 路由表 来判断

将路由表的不同条目的目的地(Destination)和子网掩码进行 与运算 结果与目标地址(服务器的)进行比较,符合的,匹配成功的网课的ip地址作为ip包头的源地址

如果前面的条目都无法匹配,则自动匹配默认网关(0.0.0.0)后续发包给路由器

完整版

网卡

网络包只是内存中的二进制数字信息,无法直接发送给对方,因此需要将数字信息转换成电信号,电信号才可以在网线上传输

负责这一过程的是网卡,控制网卡还要网卡驱动程序

网卡驱动获取网络包之后,将其复制到网卡内的缓存区,并在 开头加上报头和起始帧分界符,在末尾加上用于检测错误的帧校验序列(FCS)

  • 起始帧分界符—– 表示包起始位置的标记
  • fcs 检查包传输过程是否损坏
  • 至少含有接收方(路由器的mac地址),mac头部有什么? 待补充

最后网卡将包转换成电信号,通过网线发出

交换机

交换机的设计是将网络包 原样转发到目的地 交换机工作在mac层(网络链路层) , 也称为 二层网络设备

  • 电信号到达网线接口 , 交换机的模块 进行接受,并将电信号转换为数字信号
  • 通过FCS校验错误,没问题就放到缓冲区,此步骤和计算机网卡相同,但是工作方式不同
    计算机的网卡本身具有mac地址,可以核对包的接收方的mac地址来判断是不是发给自己的
    交换机的端口不核对,直接接收所有的包并存放到缓冲区
  • 进入缓冲区后,查询接收方的mac地址是否已经在mac地址表有记录

mac地址表的两个信息:

  • 设备的mac地址
  • 设备连接在交换机哪一个端口上

曾经发送到某mac地址的设备,当时走的端口会被记录在这个mac记录表里面

如果因为清理或者没发过,mac表内不含mac地址的信息,就会将包转发到除了源端口的所有端口,听起来不负责任,但是以太网本来就是这么设计的,将包发送到整个网络,只有接收者才接收

并且广播地址也会发给所有端口

路由器

与交换机的区别

  • 因为路由器是基于 IP 设计的，俗称三层网络设备，路由器的各个端口都具有 MAC 地址和 IP 地址；
  • 而交换机是基于以太网设计的，俗称二层网络设备，交换机的端口不具有 MAC 地址。

但是工作原理类似,都是查表判断包转发的目标

基本原理

路由器的端口有mac地址和ip地址,转发时会接受以太网包,然后从 路由表 查询转发目标, 从乡音端口发送

包接收操作

和计算机相同:检查接收方的mac地址,查看是否是自己的包,是的话放进缓冲区

查询路由表确定输出端口

完成了包接收操作之后,路由器会去掉mac头部( mac头部的作用就是将包送达路由器),其中的接收方mac地址就是路由器端口的mac地址

接下来路由器根据mac头部后方的ip头部起作用进行包的转发操作

  • 首先查询路由表判断转发目标
  • 每个条目的子网掩码和ip做与运算得到结果与目标地址进行匹配,匹配则转发,实在没有匹配则走默认路由

路由器的发送操作

根据路由表的网关列判断对方 的地址

  • 如果网关是ip地址—-此ip是转发地址, 未抵达终点,需要继续由路由器转发
  • 如果网关为空,则包里ip头部写的ip地址就是转发的目标地址,说明已经到达终点(终点路由器)

从路由器转发的过程

知道了对方的ip地址之后(从路由表网关列得知),需要通过arp协议根据ip地址查询mac地址,并根据查询结果作为接收方的mac地址,
路由器有ARP缓存,先在ARP缓存中查询,找不到就发送ARP查询请求

发送方的mac地址字段填写输出端口的mac地址,(因为原来的mac头部已经被丢弃了)

以太类型字段填写0800 表示ip协议

完成后转换成电信号通过端口(由网卡完成 发出去

网络包经过交换机到达下一个路由器

至此,数据包通过多个路由器,抵达了最后一个路由器,最后一个路由器,经过交换机,到达具体的接收方的网卡

服务端与客户端

经过上面步骤数据包抵达了服务器,开始拆解数据包

  • 首先拆mac头部,查看和自己的mac地址是否符合,符合将包收起来
  • 接着拆解ip头部,发现ip地址符合,根据ip头部里面的协议项,(06的那个十六进制数),得知自己的上层是tcp协议
  • 拆解tcp头部,获取序列包,(确定是自己需要的),是就返回一个ack,再查看里面的端口号,而接收方(这个服务器正在监听这个端口号,之前写过的, 一切始于这个服务端主动的监听某一个端口)
  • 服务器端口得到这个数据包之后,知道他应该被交给Http进程,发送过去之后,将返回的内容,(比如要访问的网页)封装在Http的响应报文里面,
  • 而相应报文再接着套上TCP,IP,MAC头部
  • 从网卡发出,经由交换机转发到路由器,调到客户端最近的路由器发给,内部的交换机,再转发给客户端,
  • 最后客户端拆解头部得到http相应报文,交给浏览器渲染
  • 最后客户端发起了四次挥手,至此双方连接断开
